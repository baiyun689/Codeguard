## 0. 项目上下文速览（给不熟悉本仓库的读者）

**Codeguard 是什么**：AI 代码审查引擎。输入 git diff，输出结构化审查问题（`Issue` 列表：severity / file / line / type / message / suggestion / confidence），覆盖安全、行为逻辑、维护质量三个维度。双语言架构：**Python = 智能编排层**（LLM 推理/编排），**Java Gateway = 护栏 + 地面真值层**（AST 提取 / 文件沙箱 / 重静态计算）。四条不变量：Python 单向调 Java；Java 绝不碰 LLM、不判断"是不是问题"；代码探索只走 Java 沙箱；不确定性只在 Python。

**当前审查核心（ADR-032 ReviewCouncil 多 Agent 编排）**，数据流：

```
START → summary（摘要+文件软路由）
      → context_provider（构造共享 ContextBundle：changed_files + summary + find_sensitive_apis + get_diff_ast）
      → 并行三发现者 Agent（ReAct）：
           ThreatModelAgent（安全，专属工具 find_sensitive_apis）
           BehaviorAgent（行为逻辑，专属工具 find_callers）
           MaintainabilityAgent（维护质量，专属工具 get_code_metrics）
           —— 三者共享 get_file_content
      → fan-in（去重 reducer）
      → council_coordinator（确定性路由，不调 LLM）
      → evidence_agent（调 Java 工具取事实 → LLM 分析证据 SUPPORTS/CONTRADICTS/INSUFFICIENT）
      → council_judge（4 条确定性规则 + 两段式去重 + 安全网 + LLM 终审，异源千问 temperature=0）
      → END
```

**关键既有资产（下文方案会反复复用）**：
- `pipeline/stages/summary.py`：一次结构化 LLM 调用，产出 `summary` / `estimated_risk_level(1~5)` / `file_focus`（文件→维度软路由）。**软路由不门控**：三审查员始终全跑，file_focus 只用于裁剪各自看到的 diff。
- `pipeline/stages/context_provider.py`：构造共享 `ContextBundle`。事实按 4000 字符截断（`_FACT_BUDGET`），AST 按文件拆块（`_split_ast_blocks`）。
- `prompts/threat-model.txt`（~220 行）/ `behavior.txt` / `maintainability.txt`：每个审查员 prompt 内嵌 ~220 行领域知识图谱（漏洞分类/缺陷模式/误报判例/判定要点），三者合计 ~690 行。
- Java 侧 `agent/ast/`：`get_diff_ast` 提取 diff 文件的完整 AST（类层次/方法签名+可见性+注解/控制流/调用边），`ASTContextFormatter` 做 LLM 文本格式化 + 两级裁剪。
- 工具沙箱：`create_tool_session(repo_path, allowed_files)` 建会话，`allowed_files` = diff 涉及文件。`get_file_content` 被 Java 沙箱限制在 `allowed_files` 内。
- 去重/裁决（reduce 端）：fan-in reducer + 4 条规则 + 两段式去重 + 安全网 + LLM 终审，已成体系。

**项目哲学（约束所有方案）**：vibe coding，"先做减法 → 先跑通再加深 → 一次只加一个变量"。尊重阶段边界（`docs/ROADMAP.md`）。**async 一直留到 chunking 阶段才切**（已明确记录的岔路口决策）。跨文件上下文（route① caller 可控 allowed_files / route② graph-RAG）于 2026-06-14 **主动搁置**。

**主力 LLM provider**：DeepSeek / 千问（qwen）系兼容端点为主，也支持 openai/claude/mock。

## 问题 1：如何设计处理大 diff

### 1.1 问题本质

大 diff 的痛点**不是"太长放不下上下文窗口"**（那只是表象），而是两条：
1. **注意力稀释**："lost in the middle"——把 3000 行 diff 塞给 LLM，埋在中间的真实问题被漏报。
2. **信噪比低**：大 diff 里常有 70%+ 是 lockfile / 生成代码 / 测试夹具 / 纯格式化，不值得花审查预算。

因此第一优先级**不是切分，是分诊（triage）**：先决定"什么值得审、审多深"，再谈"怎么切"。

### 1.2 现状钩子

- `summary.py` 已经算了 `estimated_risk_level` 和 `file_focus`——**分诊的半成品已经在了**，但目前只用于软路由裁剪，**没用于预算分配**。
- 现有架构**本质已是 map-reduce**：三审查员并行（map）→ fan-in 去重 → council_judge（reduce）。大 diff 只是把 map 的粒度从"整份 diff"变成"每个 chunk"，reduce 端几乎不用动。
- `get_diff_ast` 提供了方法/类级的结构边界。

### 1.3 方案（三层叠加，按 ROADMAP 顺序）

**第 1 层 · 噪音过滤（零成本，先做）**
在 `context_provider` 之前加确定性过滤（路径规则 + diff 内容特征），把 lockfile / 生成代码 / vendored 依赖 / 纯格式化变更降权或跳过。
- **纪律**：跳过了什么必须 log（"no silent caps"原则）——"跳过"不能看起来像"审过了"。

**第 2 层 · AST 单元切分（不是按行窗口切）**
切分的天然边界是**方法/类**，靠 `get_diff_ast`，不是"每 N 行切一刀"。
- 理由：按行切会把一个方法劈两半，语义断裂；按 AST 方法体切，每个 chunk 自包含。这是相对"裸文本 chunking"的核心优势。

**第 3 层 · 复用既有 map-reduce 骨架**
把"map 粒度"从整份 diff 改为每个 chunk。reduce 端（judge + 5 层去重）基本不用改。这是最省事的扩展路径。

**预算分配（贯穿三层）**：用 `estimated_risk_level` 给 chunk 排序，在给定 token 预算下，高风险 chunk 全量审、低风险批量审或跳过（并 log）。

### 1.4 代价 / 要盯的点

- **切分丢跨 chunk 上下文**：一个漏洞可能跨两个方法/文件。缓解：共享 `ContextBundle`（AST 骨架让每个 chunk 看到全局结构）+ 工具调用（reviewer 按需 `find_callers` 拉相邻 chunk）。
- **async 解锁点**：chunk × 3 审查员并发度上来后，同步线程池成瓶颈——**那时**切 async 才有正当理由。顺序：先同步 chunking 跑通，再切 async，一次一个变量。

### 1.5 留给深入讨论的开放问题

- 分诊的风险评分该用 `summary` 的 LLM 打分，还是加确定性启发式（diff 大小/是否碰安全 API/路径规则）？纯 LLM 打分在大 diff 下自己也会稀释。
- chunk 之间需不需要"重叠窗口"（overlap）来缓解跨 chunk 断裂？重叠会增加成本和重复报。
- 低风险 chunk"批量审"具体形态：多个 chunk 拼一个 prompt？还是降级到无工具直连（DirectEngine）？
- 大 diff 下 reduce 端去重压力剧增（候选数量 × chunk 数），5 层去重体系够不够？

## 问题 2：三路审查员 prompt 里的重知识图谱，引入什么技术处理

### 2.1 问题本质

每个审查员 prompt 内嵌 ~220 行知识图谱，三者合计 ~690 行，**静态、每次调用都重复加载**。两条痛点：
1. **成本轴**：690 行是每次审查（将来每个 chunk）的固定开销。
2. **注意力稀释轴**：diff 只碰了 SQL，模型却要读完 40 类漏洞的全量知识，真正相关的 10 行被埋没。

### 2.2 方案（三层，从零风险到高收益，叠加而非三选一）

**第一层 · Prompt Caching **
690 行知识图谱是**完全静态的前缀**。DeepSeek 有自动磁盘级上下文缓存，命中前缀约 1/10 成本，无需改代码。
- **只解决成本轴，不碰召回**，应最先做。
- 注意：这**不解决**注意力稀释——模型仍读全量。

**第二层 · 检索门控的知识卡片（解决注意力稀释）**
- 把每个审查员的知识图谱拆成**按漏洞类别的卡片**（注入卡 / 反序列化卡 / XXE 卡…）。
- 用**已经在算的信号**做检索键：`find_sensitive_apis` 扫描结果 + AST 检测到的 sink 类型 + summary。扫到 `ObjectInputStream` → 只注入反序列化卡。
- 保留**精简常驻核心**（赛道边界 + severity rubric + 高频误报判例，约 40 行），保证检索漏召时不崩。安全场景**宁可过度检索**。

> **关键判断：这里别上向量 RAG。**
> 知识库是**固定、有限、精心策展的分类法**（~40 类漏洞）。用 embedding 语义检索是杀鸡用牛刀，还引入"检索漏召"这个新失败模式。一个**确定性映射**（检测到的 sink 类别 → 知识卡片，本质是个 dict）精度更高、可测、可解释。只有当知识库变成开放式、大规模时才值得上向量库。

**第三层（可选）· 知识即工具**
审查员本就是 ReAct agent，可给它 `lookup_knowledge(category)` 工具。
- 缺点：鸡生蛋（得先有基础知识才知道该查什么）+ 多一次 round-trip。
- 倾向：**第二层的确定性注入 优于 第三层的 agent 自查**——因为能用 `find_sensitive_apis` 客观决定注入哪张卡，不依赖 LLM 自己想起来。

**明确否掉 · Fine-tuning / 蒸馏**
这个阶段不要。贵、要数据、知识会变、锁死"随时换 provider"的灵活性。

### 2.3 代价 / 要盯的点

- 拆卡片后，**知识维护**从"改一个 txt"变成"改卡片 + 维护检索映射"，复杂度上升。
- 检索键的**覆盖率**决定召回：`find_sensitive_apis` 覆盖不到的漏洞类别（如纯逻辑型鉴权缺失），卡片可能注入不到。常驻核心要兜住这类。
- Prompt caching 的**前缀稳定性**：卡片动态注入会破坏前缀一致性，削弱缓存命中——**caching 和动态卡片有张力**，需要设计"静态核心在前缀、动态卡片在后段"的布局。

### 2.4 留给深入讨论的开放问题

- 静态常驻核心 vs 动态卡片的**边界划在哪**？哪些知识"必须常驻"（漏了就崩），哪些"可按需"？
- 检索映射用什么维护？YAML 配置（sink 类别 → 卡片 id）够不够？
- 如何量化收益？需要在 `evals/` 上做 A/B：全量 prompt vs 卡片检索，看 recall 有没有回退、成本降多少。
- Prompt caching 与动态卡片的张力如何平衡（见 2.3）？

## 问题 3：AST 上下文策略——ContextProvider 只给 diff 结构，diff 外靠审查员调工具

### 3.1 当前设计

`context_provider` 急切（eager）推送 **diff 文件的 AST 骨架**进共享 `ContextBundle`；diff 之外的代码图，靠审查员在 ReAct 循环里**按需（lazy）**调工具（`get_file_content` / `find_callers`）发掘。

### 3.2 判断：原则对，但有一处结构性弱点，且恰好落在安全审查最需要的地方

**原则对在哪**：便宜的共享上下文 eager 推送，昂贵的宽上下文 lazy 拉取。diff 文件 AST 是**有界的**（知道有哪些文件）；diff 触及的传递闭包是**无界的**，eager 拉会炸上下文。所以 lazy 是对的。

**弱点在哪（核对代码确认的事实）**：

1. **lazy 拉取被沙箱卡住**：会话用 `allowed_files`（= diff 文件）创建，`get_file_content` 被 Java 沙箱限制在 diff 文件内。审查员**读不到 diff 之外文件的正文**。（这正是 2026-06-14 搁置的跨文件上下文问题。）`find_callers` 能往外看，但只返回调用方**位置/签名（元数据）**，不给正文。

2. **最有安全价值的方向恰好往外**：安全审查的核心问句是"这个被改的方法，谁调它？上游传入的参数是否被污染？"——**污染源 → 被改的 sink** 这条链，源头几乎总在 diff 之外。而 eager 推送的是 diff 文件**内部**的 AST + 内部调用边，**够不到上游调用者的污点来源**。

3. **lazy 还依赖 LLM 主动想起来调工具**：flash 类模型工具调用稳定性弱，需要看调用方才能定性的漏洞，agent 不调 `find_callers` 就直接漏报。

### 3.3 重新定义问题

真正的问题不是"eager/lazy 分得对不对"，而是：**eager 的范围绑在了"diff 边界"上，而它应该绑在"相关性"上。** 一个远处的被改文件，往往不如"被改 sink 的直接调用邻居"相关。

### 3.4 建议方向（**注意下方边界提醒——不是现在就做**）

把 eager 上下文从"diff 文件骨架"扩成 **"diff + 1 跳调用邻域骨架"**：
- 当 AST / `find_sensitive_apis` 判定某个被改方法是 sink（执行 SQL / 反序列化 / 文件操作），就把它的**直接调用方签名 + 入参来源**急切解析进 `ContextBundle`，而不是指望 agent 自己 `find_callers`。
- 这是**有界扩张**（1 跳，不是无限传递闭包），把最可预测的跨文件需求从 lazy 挪到 eager，放到 LLM 一定看得见的地方。

> **边界提醒（重要）**：跨文件上下文是 2026-06-14 **主动搁置**的（route① allowed_files 可控 / route② graph-RAG 都有问题）。上面的"1 跳邻域"**不是让现在就做**，而是当决定解封跨文件时，它是比"无脑放开沙箱读任意文件"更有原则的形状——用**调用边界**定义"1 跳"，天然有界。

### 3.5 诚实的现阶段结论

当前设计对"审 diff 本身"**没错**；它的短板在"需要 diff 外证据定性的漏洞"上，而这个短板**已知且被有意识地推迟**。应在 ADR 里写清这是已知取舍，别假装不存在；但也别现在就急着补——取决于 Codeguard 的使命边界（见开放问题）。

### 3.6 留给深入讨论的开放问题

- **使命边界**：Codeguard 是"审 diff 变更本身"（diff 引入了 bug 吗），还是要往"全程序安全分析"（整个系统可被利用吗）漂移？这决定 3.4 该不该做、做到几跳。
- "1 跳邻域"的**成本**：eager 解析调用方需要 Java 侧建全仓调用图，`get_repo_map` 已下线（ADR-020/036），得重新评估地面真值层的投入。
- 能不能用**更便宜的替代**：不 eager 拉正文，而是 eager 把"该方法有 N 个调用方，其中 M 个在 Controller 层"这类**摘要事实**注入，引导 agent 精准 `find_callers`？（介于 eager 全量与纯 lazy 之间）
- 沙箱放开 READ 到"1 跳调用邻居"的**安全边界**如何界定（Java 护栏层不变量：只读、有界）？

## 综合优先级建议

我（讨论方）的落地优先级：

1. **问题 2 第一层 · Prompt Caching**——今天就能落地，零召回风险，先拿成本收益。
2. **问题 1 · 分诊 + AST 单元切分 + 复用 map-reduce**——大 diff 是迟早要面对的，且能复用既有骨架。
3. **问题 2 第二层 · 知识卡片检索**——解决注意力稀释，但要先在 evals 上验证 recall 不回退。
4. **问题 3 · 1 跳邻域 eager 化**——留到决定解封跨文件上下文时再做，先在 ADR 记录已知取舍。

**贯穿纪律**：一次一个变量；每个方案落地前在 `DECISIONS.md` 补 ADR；在 `evals/` 上用同一数据集做前后对照，防止"优化了成本、回退了 recall"而不自知。

如果你是接手讨论的另一个模型，最值得挑战的几个点：
1. 问题 1 的"分诊优先于切分"——在你的经验里，大 diff 是分诊收益大还是切分收益大？有没有更好的第三条路？
2. 问题 2 的"固定分类法别上向量 RAG，用确定性映射"——这个判断是否成立？确定性映射的召回上限在哪？
3. 问题 3 的"eager 范围应绑相关性而非 diff 边界"——这个重构框架对不对？"1 跳调用邻域"是不是最优的相关性定义？
4. Prompt caching 与动态知识卡片的张力（2.3）——有没有两全的 prompt 布局？
