## 性能（实现是否引入难以演进的性能热点或容量隐患）

### 典型模式
在循环内做远程、数据库或序列化调用，或为方便而一次性加载无界集合：

```java
for (Order order : orders) {
    order.setCustomer(customerClient.get(order.getCustomerId()));
}
```

还包括 `findAll()` 后内存过滤、在请求线程调用阻塞外部服务、对大对象重复 JSON 转换、缓存缺失/失效策略不可见，以及每次请求重新构建昂贵客户端或正则。

### 判定要点
确认数据规模、调用频率、是否已有批量 API/缓存、以及代码是否在高流量路径。报告应说明复杂度或外部调用次数为何会随输入放大，并给出可维护的批量、分页、预取或复用方向；不能把没有基准数据的普通局部计算一律当缺陷。

### 严重级别参考
可被常规输入触发的无界内存增长、请求风暴或线程池耗尽可为 CRITICAL。明确的 N+1、热路径阻塞或明显重复 IO 通常为 WARNING。尚未证明在热路径、但会妨碍容量规划的实现，适合 INFO。

### 已知误报判例
循环对象数由严格上限限制、数据已由 repository join/batch 预取、缓存层已命中，或离线管理任务的低频操作，不应仅因循环+调用形态报性能问题。必要的审计写入也不能简单删除。

### 排除项
本标签从长期维护和容量角度评价性能结构；资源未释放属于 `RESOURCE_LIFECYCLE`，缓存读写语义错误属于 `CACHE_CONSISTENCY`，复杂分支难读但没有性能证据属于 `COMPLEXITY_CONTROL_FLOW`。
