## 可观测性与可测试性（关键行为是否可诊断、可隔离验证和安全演进）

### 典型模式
新增外部副作用却没有结构化日志、指标、审计关联或可替换依赖：

```java
public void send(Order order) {
    kafkaTemplate.send("orders", order); // 无结果处理、无 correlation id、无指标
}
```

也包括在业务方法中直接读取 `System.currentTimeMillis()`、静态调用远程客户端、吞掉异常后不记录原因、删除关键断言/日志，或把复杂逻辑写成无法注入 repository/client 的私有大方法。

### 判定要点
确认操作是否影响资金、安全、异步消息或用户可见结果；检查失败时是否能关联请求并定位原因，以及测试能否替换时钟、随机数和外部依赖。报告要指出缺少的诊断/测试接缝如何妨碍故障恢复，不能要求每个普通 getter 都加日志和指标。

### 严重级别参考
关键副作用失败后完全不可追踪、会阻碍安全/合规审计或让生产故障无法恢复时可为 CRITICAL。高价值路径缺少失败日志、指标或可测接缝通常为 WARNING。建议增加单测便利性、低风险调试信息一般为 INFO。

### 已知误报判例
统一 AOP/SDK 已自动记录 trace、metric 和审计，框架集成测试可覆盖真实依赖，或日志会暴露敏感数据而被有意省略时，不应机械要求额外日志。已有上层重试/死信监控也要纳入判断。

### 排除项
日志内容泄露属于 `DATA_EXPOSURE`，消息投递语义属于 `MESSAGE_DELIVERY`，异常控制流正确性属于 `ERROR_HANDLING`。本标签关注诊断与验证能力，而非替代这些运行时正确性审查。
