## 重复与设计（重复逻辑和错误抽象是否增加一致性维护成本）

### 典型模式
多个 Controller/Service 复制相同的参数校验、实体映射、重试、状态判断或异常转换，只替换一个字段：

```java
if (request.getId() == null) throw new BadRequestException("id required");
// 在多个 endpoint 中重复相同校验、查询和映射
```

另一种模式是为了消除少量重复而创建泛化过度的 `BaseService<T, R, C>`，调用方需要理解大量类型参数和布尔开关才能完成简单操作。

### 判定要点
确认重复是否代表同一业务规则、是否已经开始漂移、未来是否必须同步修改；建议抽取应保留明确领域名称和边界。不要把偶然相似的两三行、或尚未证明稳定的代码过早抽象为公共框架。

### 严重级别参考
多处重复已经漂移、会使规则修复遗漏或测试成本显著上升时通常为 WARNING。局部样板或只影响可读性的重复为 INFO。若复制规则已经产生安全、资金或状态错误，应由 ThreatModelAgent 或 BehaviorAgent 报告已发生的领域缺陷，本标签只报告导致漂移的设计根因。

### 已知误报判例
不同业务流程虽然代码相似但演进节奏不同、DTO 显式映射为防止隐式字段泄露、或框架约定要求各 handler 保持独立，不能强行抽象。已经由共享 validator、mapper 或模板方法覆盖的逻辑也不是重复。

### 排除项
本标签不评价算法性能或控制流本身；N+1 属于 `PERFORMANCE`，过深分支属于 `COMPLEXITY_CONTROL_FLOW`，外部 API 行为不一致则是 `API_CONTRACT`。
