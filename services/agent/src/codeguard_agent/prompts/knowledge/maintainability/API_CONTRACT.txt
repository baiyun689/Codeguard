## API 契约（公共接口的输入、输出、错误和兼容性约定是否清晰稳定）

### 典型模式
修改 Spring MVC endpoint、service interface、DTO 或事件 payload 时，参数含义和返回约定隐含在实现中：

```java
public UserDto findUser(String id) {
    return repository.findById(id).orElse(null); // 调用方原先依赖 404/异常
}
```

典型风险还包括把字段从必填改可空、改变分页/排序默认值、重用内部实体作为 JSON 响应、以魔法字符串表示状态，或删除已有错误码而未更新调用方。

### 判定要点
确认接口是否被跨模块、客户端、消息消费者或测试使用；比较变更前后的空值、异常、HTTP 状态、字段名、单位、顺序和默认行为。问题应是契约模糊、破坏兼容或让新增调用者难以正确使用，而非单纯偏好某种命名风格。

### 严重级别参考
会导致广泛客户端错误、错误写入或不可恢复的协议不兼容可为 CRITICAL。影响已知调用方、但可通过版本化或迁移修复的兼容性变化通常为 WARNING。文档、命名或局部约束不够明确但未见实际破坏时为 INFO。

### 已知误报判例
明确标记为内部 package-private API、已通过 `/v2` 版本端点隔离、同一提交同步更新全部消费者，或由 OpenAPI/JSON Schema 和契约测试强制约束的字段变更，不能仅凭签名变化报问题。

### 排除项
认证/授权策略属于安全或业务权限审查；空指针和非法状态的运行时防护属于 `NULL_STATE_SAFETY`；数据库事务边界属于 `TRANSACTION_ATOMICITY`。本标签只评价接口契约的可理解性、稳定性与演进成本。
