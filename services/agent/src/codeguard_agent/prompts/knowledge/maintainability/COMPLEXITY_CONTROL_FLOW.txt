## 控制流复杂度（分支、异常和状态转换是否仍可被安全理解和修改）

### 典型模式
一个 Service 方法同时处理校验、授权、状态机、持久化和补偿，嵌套分支掩盖核心路径：

```java
if (a) { if (b) { try { if (c || d) { ... } } catch (...) { ... } } }
```

也包括多处 `return` 改变清理顺序、switch 漏掉枚举分支、布尔参数控制多种业务模式，以及 catch 后继续执行造成控制流难追踪。

### 判定要点
判断新增复杂度是否跨越清晰的业务边界、是否导致状态转换或副作用难以枚举、是否已有可提取的策略/守卫条件。只在复杂度已经影响正确理解、测试覆盖或后续修改风险时报告，并定位具体难以维护的路径。

### 严重级别参考
隐藏错误状态转换、错误继续执行并可能造成数据破坏时可为 CRITICAL。多层嵌套导致关键分支难审查、很可能遗漏修改的通常为 WARNING。可读性下降但路径仍直接、可通过轻量重构改善的为 INFO。

### 已知误报判例
安全校验或协议解析需要显式分支、短小 switch 已穷尽枚举、生成代码，及已有清晰注释和分支单测覆盖的状态机，不能只按 if 数量报问题。

### 排除项
重复实现应归入 `DUPLICATION_DESIGN`；异常是否被正确处理属于 `ERROR_HANDLING`；业务并发一致性属于 `CONCURRENCY_CONSISTENCY`。本标签聚焦代码形状造成的维护风险。
